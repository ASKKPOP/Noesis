# Phase 7 — UI Design Contract

> Visual and interaction contract for the Peer Dialogue → Telos Refinement surfaces: the panel-level `↻ refined via dialogue` badge on the Inspector's TelosSection, the `useRefinedTelosHistory` client hook, and the firehose `dialogue_id` filter chip. Generated by gsd-ui-researcher (`--auto`), verified by gsd-ui-checker.

**Design posture:** Phase 7 extends the Phase 3 / Phase 6 operator dashboard (`/grid`). The dominant visual voice is unchanged — terse, data-dense, Honeycomb/Grafana. The new surfaces are **Brain-initiated**, so their color tone is deliberately distinct from Phase 6's tier palette (H2 blue / H3 amber / H4 red reserved for operator-forced actions). Peer dialogue refinement is cognition-internal; the badge must read as *informational* (a trace of the Nous's own choice), not as an alert or an operator action.

**Philosophical anchor (PHILOSOPHY §1 + D-18):** UI NEVER displays the plaintext `new_goals`. The dashboard's hash-only boundary is the badge + dialogue_id click-through. Brain owns goal text; Grid (and therefore the dashboard) sees only cryptographic diffs.

**New in Phase 7 vs. Phase 6:**
1. `<TelosRefinedBadge />` — panel-level chip on `TelosSection` (one per Nous, not per-goal — D-30).
2. `useRefinedTelosHistory(did)` — client-side firehose filter hook; zero new network traffic (D-28).
3. `<DialogueFilterChip />` — active-filter chip in firehose when `?firehose_filter=dialogue_id:<16-hex>` is set (D-29).
4. `useFirehoseFilter()` — thin query-param reader/clearer bound to Next.js `useSearchParams` + `useRouter`.

## Requirement Mapping

| Requirement | Surface | Contract |
|-------------|---------|----------|
| **DIALOG-01** — Dialogue aggregation + `dialogue_context` widening | (no UI surface — Grid-internal) | Phase 7 UI does not render dialogue_context; it renders only *downstream* `telos.refined` audit events |
| **DIALOG-02** — `telos.refined` hash-only event | Consumed indirectly via `useRefinedTelosHistory` | Hook subscribes to the firehose stream (existing WS surface), filters `eventType === 'telos.refined' && payload.did === did`, collects `triggered_by_dialogue_id` + `after_goal_hash` |
| **DIALOG-03** — Inspector badge + firehose link | `<TelosRefinedBadge />` + `<DialogueFilterChip />` | Badge visible iff refinement history is non-empty for the selected Nous; click-through navigates to `?firehose_filter=dialogue_id:<most-recent-triggered_by_dialogue_id>` |

**Philosophical constraint (PHILOSOPHY §1, D-18):** `new_goals` plaintext NEVER enters any dashboard component. The hook surface is `{ lastRefinedDialogueId: string | null, refinedCount: number, refinedAfterHashes: ReadonlySet<string> }` — no goal text ever. The UI renders only counts and dialogue_id references.

## Color

Dark theme base. 60/30/10 distribution preserved at page level.

| Role | Value | Usage in Phase 7 |
|------|-------|-------|
| Dominant (60%) | `#0A0A0B` | Page background |
| Secondary (30%) | `#17181C` | Badge background, filter-chip background |
| Tertiary (~8%) | `#23252B` | Badge hover background, filter-chip hover, clear button hover |
| Accent (10%) | `#7DD3FC` (sky-300) | Focus rings only — 2px outline + 2px offset, same as Phase 6 |
| **Dialogue (new)** | `#818CF8` (indigo-400) | **Badge border + icon + label text; filter-chip border** |

### The `dialogue` color — why indigo-400 and not amber/red/cyan

The badge must be **visually distinct from the H2/H3/H4 tier palette** so operators do not confuse a Nous-initiated refinement with an operator-forced action.

| Candidate | Why rejected |
|-----------|--------------|
| sky-300 `#7DD3FC` | Reserved for focus rings (Phase 3 discipline). Using it for a filled chip breaks the "accent = focus only" rule. |
| blue-400 `#60A5FA` | Reserved for H2 Reviewer (Phase 6). Reusing would conflate operator-logged read with Brain-initiated refinement. |
| amber-300 `#FCD34D` | Reserved for H3 Partner (Phase 6). Reusing suggests operator cautionary action. |
| red-400 `#F87171` | Reserved for H4 Driver + destructive state. Reusing is a philosophical error — refinement is *not* destructive. |
| green-400 `#4ADE80` | Reserved for `--color-success`; semantically implies "this is good." Grid does not judge refinement quality (REV-04 invariant, D-30 echoed). |
| cyan-400 `#22D3EE` | Too close to sky-300 focus ring; fails distinguishability at small chip size. |
| **indigo-400 `#818CF8`** | **Not used elsewhere in the dashboard.** Neutral-positive, cognitive-adjacent (indigo historically associated with thought/reflection). Sufficient contrast on `#17181C` background. |

**Contrast verification (checker must confirm):**
- Indigo-400 `#818CF8` on `#17181C` → ~6.4:1 (AA PASS for 14px; AA PASS for small text per WCAG 1.4.3).
- Indigo-400 used for 1px border + label text (`#818CF8`) on background `#17181C` → same 6.4:1.
- Badge count parens `(3)` in `#818CF8` on `#17181C` → 6.4:1 (AA for 12px 500-weight).

### Accent reserved for (Phase 7 additions):
1. Focus ring on `telos-refined-badge-trigger` button and `firehose-filter-clear` button — 2px sky-300 outline + 2px offset.
2. **No other uses.** Accent is NEVER used for the badge fill, border, or label; those are all `#818CF8`.

### Color-independent redundancy (accessibility):
- Badge carries a `↻` glyph (U+21BB CLOCKWISE OPEN CIRCLE ARROW) — conveys "refined/re-derived" for colorblind users.
- Badge `aria-label` carries the full meaning: `"Telos refined via peer dialogue — 3 refinements in history. Click to view triggering dialogue in firehose."`
- Filter chip carries literal text `dialogue_id:` — no user depends on color alone to understand the filter.

### Voice guidelines (Phase 7 additions to Phase 3/6 list)

- Never say "refreshed" — the word is `refined`, from PHILOSOPHY §1 and REQUIREMENTS DIALOG-02. `refreshed` reads as a cache-invalidation UX; `refined` carries the cognitive-update meaning.
- Never say "updated goals" or "new goals" in any user-visible string — both imply that the dashboard knows what changed. The dashboard knows only the *fact* of refinement (a hash diff), not the content.
- Never say "auto-refined" or "automatically" — refinement is Brain's sovereign choice (PHILOSOPHY §1). Prefer `via dialogue`, which names the trigger without implying automation.
- The word `dialogue` (British spelling) matches REQUIREMENTS.md DIALOG-01; never use `dialog` (American spelling) — that word is reserved for the Phase 6 `<ElevationDialog />` UI primitive, and conflating them invites confusion.

### Empty / error / edge states

| Element | Copy |
|---------|------|
| Primary CTA (phase-level) | **Not applicable.** Phase 7 has no primary CTA — the badge is a navigation affordance, not an action. |
| Empty state (no refinements) | **Not applicable.** Badge is simply absent. TelosSection's existing `"No active goals. Telos is quiescent."` empty state (Phase 3) is unchanged. |
| Filter with zero matching events | Firehose shows: heading `No matching events for dialogue_id {value}. Press × to clear.` (14/400 Inter, neutral-400). Replaces event list entirely. |
| Error state (hook receives malformed event) | **Silent drop at hook boundary.** Malformed `telos.refined` events (missing `triggered_by_dialogue_id`, non-16-hex, missing `after_goal_hash`, etc.) are filtered out by `useRefinedTelosHistory`. No error surface to user — a malformed event at the boundary means the Grid producer gate failed, which is a dev-visible issue, not an operator-visible one. |
| Destructive confirmation | **Not applicable.** Phase 7 has no destructive actions. The badge is navigation-only. |

## Interaction Contract

Three new interactions in Phase 7. Documenting them so the planner does not invent more.

| Interaction | Trigger | Response | Timing |
|-------------|---------|----------|--------|
| Click TelosRefinedBadge | Click `telos-refined-badge-trigger` button OR press Enter/Space while focused | `router.push(...)` with current URL's query params updated to `?firehose_filter=dialogue_id:<lastRefinedDialogueId>` (preserves existing `?tab=` if present). If currently on `tab=firehose`, stays on tab; if on `tab=economy`, switches to `tab=firehose`. | <16ms (client-side navigation) |
| Click DialogueFilterChip clear button | Click `firehose-filter-clear` OR press Enter/Space while focused | `router.push(...)` removing `firehose_filter` param; all other query params preserved. Firehose re-renders with no filter; chip unmounts. | <16ms |
| Keyboard navigation to badge | Tab through TelosSection | Badge receives focus after the last goal's priority chip; focus ring (sky-300) visible; Enter/Space activates. | native |

### Badge click-through URL contract (locked)

- **Input:** `lastRefinedDialogueId` (16-hex) from `useRefinedTelosHistory`.
- **Output URL:** same pathname + `?<preserved-params>&tab=firehose&firehose_filter=dialogue_id:<16-hex>`.
- **Query-param key:** `firehose_filter` (matches orchestrator spec D-29).
- **Value format:** `<key>:<value>` where `<key>` is one of a closed, validated set — Phase 7 introduces only `dialogue_id`. Unknown keys are silently ignored by `useFirehoseFilter` (future-proof for `goal_hash:` or `did:` filters without schema break).
- **Value regex at parse time:** for `dialogue_id`, `/^[0-9a-f]{16}$/` — malformed values are ignored (filter not applied, chip not rendered). Mirrors the producer-boundary regex discipline from Phase 6 D-19 and Phase 7 D-31.

### Firehose filter application (locked: **dim, not hide**)

- Non-matching rows render at `opacity-40` with `pointer-events: none` on hover affordances. Row text remains selectable for copy-paste debugging.
- Matching rows render at full opacity (opacity-100).
- **Rationale for dim over hide:** the firehose is a debugger's surface. Hiding non-matching events would make it impossible to see the *temporal context* of the dialogue (what events preceded and followed the refinement). Dimming preserves context while drawing the eye to matches. Matches Phase 3's existing debug-surface voice.
- Escape hatch: clicking × on the filter chip restores all rows to full opacity in one frame.

### Keyboard accessibility baseline (required for checker)

- Badge is a `<button type="button">` element. Tab-ordered after the last goal's priority chip inside the TelosSection (preserves reading order).
- Filter chip clear button is a `<button type="button">`. Tab-ordered as the first focusable in the firehose panel when the chip is mounted.
- Both buttons display the sky-300 focus ring identical to Phase 6's pattern: `focus-visible:outline focus-visible:outline-2 focus-visible:outline-offset-2 focus-visible:outline-sky-300`.
- `prefers-reduced-motion: reduce` — no new animations introduced, so no adaptation needed. Filter-chip mount/unmount is instant (no fade).

## State Contract

| State | Behavior |
|-------|----------|
| **SSR / initial render** | Badge NOT rendered server-side. `useRefinedTelosHistory` is client-only (subscribes to WebSocket); on first client render the hook returns `{ lastRefinedDialogueId: null, refinedCount: 0, refinedAfterHashes: new Set() }`. Badge appears on the client once firehose events arrive. This matches Phase 3's pattern for any firehose-dependent UI (no flash of stale state because the badge simply isn't in the SSR DOM). |
| **Loading** | No explicit loading UI. During firehose connection bootstrap (Phase 3's `useFirehose` reconnect logic), the hook returns `refinedCount: 0` — badge stays absent. No skeleton, no spinner. Phase 3 voice: empty state is the honest state. |
| **Empty (refinedCount === 0)** | Badge not rendered. No placeholder, no "no refinements yet" copy. Absence is the signal. |
| **Populated (refinedCount ≥ 1)** | Badge rendered with current count. `lastRefinedDialogueId` updates reactively as new `telos.refined` events stream in. |
| **Filter active but 0 matches** | Firehose panel shows the filter chip + the empty-match heading `No matching events for dialogue_id {value}. Press × to clear.` No rows rendered. |
| **Filter active, all rows match** | Firehose panel shows the filter chip + all rows at full opacity (functionally equivalent to no filter, but chip remains mounted so operator sees the applied filter). |
| **Filter param malformed** | `useFirehoseFilter` returns `filter: null`. Chip not mounted. Firehose renders unfiltered. No error toast (malformed URL is typically a bookmark typo, not a bug). |
| **Nous `did` prop is null** (Inspector not attached to any Nous) | Hook returns `refinedCount: 0`; badge not rendered. |
| **Nous switched in Inspector** | Hook re-derives from the firehose history for the new `did`. History is client-side — switching is instant (no refetch). Badge count updates immediately. |
| **Firehose disconnection mid-session** | Existing `useFirehose` reconnect UX applies. Accumulated refinement history persists in hook state across reconnect (no flush on disconnect). On reconnect, any events that arrived while disconnected are delivered via the existing backlog-replay mechanism (Phase 3 contract). |

**Hook internal state:** `refinedAfterHashes` is retained on the hook surface for future per-goal attribution (D-30 deferral exit path) — if Brain later exposes per-goal hashes, TelosSection can consume this set without a hook-signature change.

## Auto-Resolved Design Questions

Because Phase 7 was dispatched in `--auto` mode, the following questions were resolved by picking the most conservative option matching Phase 3/6 precedent. Each is logged for checker scrutiny.

| Question | Resolution | Rationale |
|----------|------------|-----------|
| Firehose file path: orchestrator suggested `dashboard/src/app/firehose/...`, but Phase 3 has no such route. | **Used `dashboard/src/app/grid/components/firehose.tsx`** and added `firehose-filter-chip.tsx` alongside it. | Matching existing path avoids an unrelated restructure. The `firehose_filter` URL param still lives on the `/grid` route (where the firehose panel is mounted), consistent with D-29. |
| Filter behavior when active: **dim non-matching** vs. **hide non-matching**. | **Dim at `opacity-40`.** | Preserves temporal context for debugging — the firehose is a debugger's surface (Phase 3 voice). Hiding loses the "what happened before the refinement" signal. |
| Badge color: any of sky / blue / amber / red / green / cyan / indigo / violet. | **Indigo-400 `#818CF8`.** | Only unused spectrum slot that passes contrast on `#17181C`, is not claimed by another semantic (tier, focus, success, warning, destructive), and reads as "cognitive / reflective." |
| Badge weight: 500 vs. 600. | **500.** | Phase 6 reserves 600 for load-bearing tier chips. Refinement is informational, not authority — 500 signals "notice, not act." |
| Badge glyph: `↻` (U+21BB) vs. `⟳` (U+27F3) vs. Lucide `<RefreshCw />` icon vs. none. | **`↻` unicode glyph.** | Phase 3/6 precedent: monochrome Lucide icons are used sparingly and only where semantics demand an icon (not a glyph). `↻` is a single-codepoint typographic glyph, renders inline with text at 12px without extra import, and conveys "refined/re-derived" without ambiguity. |

| Badge placement in TelosSection: heading-row vs. above heading vs. below goals. | **Heading-row right-aligned** (flex justify-between with the `Telos` heading). | Mirrors the Phase 3 inspector section-heading pattern; preserves vertical density. |
