<objective>
Close the three Grid-side gaps that block the dashboard before a single line of frontend code is written:

1. CORS allowlist — the dashboard is served from http://localhost:3001 and must call the Grid at http://localhost:8080. Right now `grid/src/api/server.ts` does NOT register `@fastify/cors`, so every browser fetch would fail the preflight. `@fastify/cors@^10.0.0` is already in `grid/package.json` dependencies but unused.
2. Regions endpoint — `GET /api/v1/grid/regions` currently returns `{ regions }`. The region map needs edges, and `SpatialMap` already has `getConnections(regionId)` but there is no aggregate endpoint exposing the full connection list. Extend the response shape in-place to `{ regions, connections }` rather than add a sibling route (the dashboard needs both together).
3. Tick audit emission — `tick` is on the broadcast allowlist (`grid/src/audit/broadcast-allowlist.ts:33`) but NO code in the Grid ever calls `audit.append('tick', ...)`. `WorldClock.advance()` fires `onTick` listeners but does not write to the chain. The HeartbeatStore cannot rely on non-tick events to know the tick rate, so the dashboard either falls back to `/api/v1/grid/clock` every poll or we fix this at the source. We fix it at the source.

Purpose: Without this plan, Plans 03-05 produce a dashboard that either CORS-fails on first fetch, cannot draw region edges, or cannot show a heartbeat.

Output:
- `@fastify/cors` registered with a dev allowlist in `buildServerWithHub`.
- `GET /api/v1/grid/regions` returning `{ regions, connections }`.
- A single `clock.onTick` listener in `GenesisLauncher.bootstrap` that appends a `tick` audit entry on every tick.
- Three new Vitest files proving each of the three behaviors.
- All 289 existing grid tests still green.
</objective>

MySQL store already persists allowlisted events; sizing is a Phase 4 concern. |
