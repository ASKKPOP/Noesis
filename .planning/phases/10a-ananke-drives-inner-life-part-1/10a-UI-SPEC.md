## Scope Boundary

Phase 10a ships **exactly one UI surface**: a new section component
`<AnankeSection did={did} />` mounted inside the Inspector Overview tab as the
5th sibling in render order — between Thymos and Telos. It is not a new page,
route, modal, or design-system shift. No new global tokens, no new primitives,
no new dependencies.

**In scope:**
- Panel layout within `inspector.tsx` Overview tab
- 5 drive rows × 3 level buckets × 2 direction indicators (rising/falling) rendered as Unicode + color
- Accessibility contract: per-drive aria-label matrix; color-not-sole-channel compliance
- Baseline-level initial state when no `ananke.drive_crossed` has landed for the DID
- Zero-numerics guarantee (DRIVE-05 enforcement at the render surface)
- Source-of-truth contract: panel reads from the broadcast chain, never from a new Brain RPC or private state endpoint

**Out of scope (deferred or anti-feature):**
- Numeric drive-float rendering anywhere in DOM, tooltip, title, or aria text (anti-feature per DRIVE-05 + D-10a-07)
- Drive time-series charts / sparklines (roadmap defers quantitative views to v2.3)
- Drive → Telos / action coupling UI (drives are advisory only per D-10a-06; no UI surface implies a drive "caused" an action)
- Dashboard Drives page / tab (10a is Inspector-panel-only; a top-level page is a future decision)
- New route, new entry in `DASHBOARD_ROUTES` E2E SC#1 coverage list
- Mobile / responsive adaptations (dashboard is desktop-only per existing convention)
- New icon library, new SVG assets, or new third-party components

## Panel Layout

### Position in the Inspector drawer

```
Inspector drawer (fixed right, 384px wide, dark neutral-950)
├── header: "Nous Inspector" + close
├── tab strip: [ Overview | Relationships ]
│
├── Overview tabpanel (hidden when tab=relationships)
│     ├── <PsycheSection>       ── Phase 4 / v2.0
│     ├── <ThymosSection>       ── Phase 4 / v2.0
│     ├── <AnankeSection>  NEW  ── Phase 10a ◄◄◄ THIS PHASE
│     ├── <TelosSection>        ── Phase 7 refinement badge
│     └── <MemorySection>       ── Phase 4 / v2.0
│
└── Relationships tabpanel (Phase 9; unchanged by 10a)
      └── <RelationshipsSection>
```

**Mount position rationale:**
- Psyche (persistent personality) → Thymos (current mood) → **Ananke (current drives)** → Telos (current goals) → Memory (recent history) forms a coherent inner-life-outward ordering: personality → emotion → drive pressure → goals → history.
- Drives sit between Thymos and Telos because drives are the mechanism that couples present state (Thymos) to future orientation (Telos) — even though the coupling itself is advisory (D-10a-06), the visual adjacency makes the relationship legible without implying coercion.
- The position choice is a layout convention, not a semantic claim. A future phase can reorder without breaking this spec.

### Section shell (identical to Psyche/Thymos/Telos siblings)

```jsx
<section
    data-testid="section-ananke"
    aria-labelledby="section-ananke-title"
    className="mb-4"
>
    <h3
        id="section-ananke-title"
        className="mb-2 text-sm font-semibold text-neutral-100"
    >
        Drives
    </h3>
    <ul role="list" aria-label="Current drive pressure levels" className="flex flex-col gap-1">
        {/* 5 rows, ordered by DRIVE_ORDER constant */}
    </ul>
</section>
```

### Row shape (one per drive, five rows total)

```
┌────────────────────────────────────────────────────────────────────────┐
│ [●]  🪿 hunger          med         ↑                                 │
│  │    │       │           │          │                                 │
│  │    │       │           │          └─ direction glyph (only when    │
│  │    │       │           │             transitioning; absent at rest)│
│  │    │       │           └────────────── level bucket label (text)   │
│  │    │       └────────────────────────── drive label                  │
│  │    └──────────────────────────────── drive glyph (Unicode)         │
│  └──────────────────────────────────── level-colored dot              │
└────────────────────────────────────────────────────────────────────────┘
```

Concrete row JSX contract:

```jsx
<li
    data-testid={`drive-row-${drive}`}
    data-drive={drive}
    data-level={level}
    data-direction={direction ?? 'stable'}
    className="flex items-center gap-2 rounded border border-neutral-800 bg-neutral-900 px-2 py-1"
>
    <span
        data-testid={`drive-dot-${drive}`}
        aria-hidden="true"
        className={`inline-block h-2 w-2 rounded-full ${LEVEL_STYLE[level].dotClass}`}
    />
    <span
        data-testid={`drive-glyph-${drive}`}
        aria-hidden="true"
        className="text-sm leading-none"
    >
        {DRIVE_GLYPH[drive]}
    </span>
    <span className="flex-1 text-xs text-neutral-200">
        {drive}
    </span>
    <span
        data-testid={`drive-level-${drive}`}
        className={`text-xs ${LEVEL_STYLE[level].textClass}`}
        aria-label={`${drive} level ${level}${direction ? `, ${direction}` : ''}`}
    >
        {level}
    </span>
    {direction && (
        <span
            data-testid={`drive-direction-${drive}`}
            aria-hidden="true"
            className={`text-xs tabular-nums ${LEVEL_STYLE[level].textClass}`}
        >
            {DIRECTION_GLYPH[direction]}
        </span>
    )}
</li>
```

**Row height:** natural flow (≈24px); no `firehose-row` exception. Gap between
rows is `gap-1` (4px). Total panel height ≈ `5 rows × 24px + 4 × 4px gap + h3` ≈ 140px.

**Drive ordering (locked):** `DRIVE_ORDER = ['hunger', 'curiosity', 'safety', 'boredom', 'loneliness']`. Matches REQUIREMENTS.md DRIVE-01 enumeration. Order is stable across ticks regardless of which drives have recently crossed — re-ordering by recency would leak a latent-pressure signal and is an anti-feature.

## Drive Glyph Matrix

Unicode glyphs only. Each glyph is a single code point selected to be:
- distinguishable from the others at 14px/`text-sm` with `leading-none`
- legible on `neutral-900` background
- semantically suggestive of the drive without depending on color
- present in the system default emoji fonts across macOS / Linux / Windows

| Drive | Glyph | Code point | Semantic cue | Fallback text (if glyph fails to render) |
|-------|-------|-----------|--------------|------------------------------------------|
| hunger | ⊘ | U+2298 (CIRCLED DIVISION SLASH) | empty vessel / unmet need | `hunger` |
| curiosity | ✦ | U+2726 (BLACK FOUR POINTED STAR) | attention / novelty pull | `curiosity` |
| safety | ◆ | U+25C6 (BLACK DIAMOND) | shelter / stability shape | `safety` |
| boredom | ◯ | U+25EF (LARGE CIRCLE) | hollow / stimulus deficit | `boredom` |
| loneliness | ❍ | U+274D (SHADOWED WHITE CIRCLE) | isolated / two-not-one absence | `loneliness` |

**Font-fallback discipline:** the glyphs above are in the Unicode BMP and
monochrome-only (no emoji variation selector VS-16). On systems where Inter or
the default sans stack does not carry a glyph, the browser falls back to the
system symbol font; no glyph should resolve to the Unicode replacement character
(`U+FFFD `). Planner executes a visual-smoke screenshot on macOS + Linux +
Windows during verification; if any glyph renders as a tofu box, swap the
specific glyph for its fallback text (the drive name in `text-xs`) via a named
conditional in `DRIVE_GLYPH` (document the swap in the phase SUMMARY).

**No color channel on the drive glyph.** The glyph sits in `text-neutral-200` at
all levels. Level encoding is handled exclusively by the dot + text-label color
pair. This is a deliberate double-encoding: shape (drive) and color (level) are
orthogonal axes so color-blind and low-vision operators still distinguish both.

## Level Bucket Color Encoding

Clones the Phase 9 three-bucket warmth palette at the Tailwind-token tier. No
new globals.css tokens are introduced. Values re-use the neutral-amber-rose ramp
Phase 9 established for warmth, rebound to drive semantics.

| Level | Semantic | Dot class | Text class | Rationale |
|-------|----------|-----------|-----------|-----------|
| `low` | baseline / quiescent | `bg-neutral-400` (#9CA3AF) | `text-neutral-400` | Matches Phase 9 `cold` bucket — neutral conveys "nothing to act on" |
| `med` | elevated / active | `bg-amber-400` (#FBBF24) | `text-amber-400` | Matches Phase 9 `warm` bucket — amber conveys "pay attention but not alarm" |
| `high` | pressing / salient | `bg-rose-400` (#FB7185) | `text-rose-400` | Matches Phase 9 `hot` bucket — rose conveys "this drive is loud right now" |

**Palette reuse is intentional, not aesthetic.** Phase 9 warmth and Phase 10a
drive pressure are semantically adjacent — both are three-bucket ordinal scales
over a derived inner-state dimension rendered inside the same Inspector drawer.
Using the same palette lets the operator's visual vocabulary transfer across
tabs without re-learning. The two scales are disambiguated by the section
heading and the drive glyph, not by color.

**Contrast compliance:**
- `text-amber-400` on `bg-neutral-900` → 9.8:1 (AAA large, AA small)
- `text-rose-400` on `bg-neutral-900` → 6.9:1 (AA large, AA small)
- `text-neutral-400` on `bg-neutral-900` → 5.7:1 (AA large, AA small)

All three meet WCAG 2.1 AA for small text. Phase 9 human-verify checkpoint
validated the same tokens in the real dark theme under Plan 09-05 — no new
human-verify gate is required for Phase 10a.

**MUST NOT reuse accent for anything else in 10a.** The `accent` token (#7DD3FC)
remains reserved for the active tab underline and H5 modal link hover (Phase 9).
Drives are not accent-highlighted; they are level-graded on their own scale.

## Direction Indicator

When a drive transitions from one level to another, the row carries a direction
glyph for a window long enough to be observable but short enough that the panel
doesn't accumulate stale transition state.

| Direction | Glyph | Code point | Color | Aria |
|-----------|-------|-----------|-------|------|
| rising | ↑ | U+2191 (UPWARDS ARROW) | inherits `LEVEL_STYLE[level].textClass` | part of combined row aria-label (below); glyph itself is `aria-hidden` |
| falling | ↓ | U+2193 (DOWNWARDS ARROW) | inherits `LEVEL_STYLE[level].textClass` | same |
| stable (no glyph) | — | — | — | aria-label omits direction clause |

**Display window:** the direction glyph is rendered while the row's level
matches the `level` field of the most recent `ananke.drive_crossed` event for
that (DID, drive) pair — and no newer event of any kind has shifted that level
off. Concretely: the glyph is visible from the moment the transition audit
entry lands until **either** (a) another `ananke.drive_crossed` for the same
(DID, drive) supersedes it, or (b) the Inspector drawer closes and reopens on a
different DID.

**No wall-clock timers.** The direction indicator is not on a time-based fade —
there is no `setTimeout`. Its visibility is a pure function of the audit stream,
so the panel stays deterministic under replay (T-09-03-adjacent: no Brain-side
wall-clock read, no dashboard-side wall-clock read for semantic state).

**Animation contract:** **no animation.** Icon swaps are instantaneous when a
new audit entry lands. The existing AgencyIndicator (Phase 6) changes its Chip
color on tier swap without animation, and that precedent is the one this phase
extends. Transition smoothing is an anti-feature for an observability surface —
operators must see exact-tick semantic changes, not a fade that hides the edge.

## Baseline / Initial State (First-paint contract)

When the Inspector opens on a DID for which **no** `ananke.drive_crossed` event
has yet landed in the audit chain (fresh Nous just past `bios.birth`, or a
replay before any crossing), the panel renders five rows at their **per-drive
baseline level**.

**Baseline levels MUST be deterministic at read time**, derived from the same
per-drive baseline constants the Brain uses. The dashboard does not fetch the
baseline vector — it **mirrors the constants** from the Brain (`brain/src/ananke/config` or equivalent; exact path locked by planner) into a
TypeScript constant at `dashboard/src/lib/protocol/ananke-types.ts` using the
same SYNC-header drift-detector pattern v2.1 Phase 6 established for
`agency-types.ts` (see STATE.md Accumulated Context Plan 06-02: "Dashboard type
mirror pattern — hand-copied dashboard mirrors of grid protocol types. SYNC
header + drift-detector test (fs.readFileSync) in place.").

**Baseline mirror format (locked in planner):**

```ts
// dashboard/src/lib/protocol/ananke-types.ts
// SYNC: mirrors brain/src/ananke/config.py DRIVE_BASELINES
// Drift is detected by dashboard/test/lib/ananke-types.drift.test.ts
// which reads the Python source and fails if values diverge.
export const DRIVE_BASELINE_LEVEL: Record<DriveName, DriveLevel> = {
    hunger: 'low',      // mirrors baseline value, bucketed
    curiosity: 'med',   // mirrors baseline value, bucketed
    safety: 'low',      // mirrors baseline value, bucketed
    boredom: 'med',     // mirrors baseline value, bucketed
    loneliness: 'med',  // mirrors baseline value, bucketed
};
```

**The dashboard never stores or renders the raw baseline float.** It stores
only the bucketed `DriveLevel` computed by the Brain (or by an exact-match
mirror of the Brain's bucketing function over the known baseline floats). The
baseline floats themselves never enter the dashboard bundle — grep test over
`dashboard/src/**` asserts no literal `0.3 | 0.5 | 0.2 | 0.4` that would match
a drive baseline (exact float enumeration and the associated grep pattern are
planner's decision, in line with D-10a-07 DRIVE_FORBIDDEN_KEYS).

**Baseline rows render without a direction glyph.** Baseline is a state, not a
transition — the row reads `"curiosity  med"` with no arrow. This is the
deterministic "current tier read-out" required by the phase spec — never a
"waiting for event" placeholder, never an empty state, never a spinner.

**If the Brain's baseline constants change in a future phase**, the drift-
detector test fails and the dashboard mirror must be updated in the same
commit per the CLAUDE.md doc-sync rule.

## Source-of-Truth Contract

The Drives panel **reads exclusively from the existing dashboard audit
subscription** — the same WebSocket + presence-store + store pattern that
serves the firehose, AgencyIndicator, and Phase 9 relationships surface.

| Read | Where from | Why |
|------|-----------|-----|
| current level per (DID, drive) | most recent `ananke.drive_crossed` event in the local audit buffer where `actorDid === did && payload.drive === drive`; fallback to `DRIVE_BASELINE_LEVEL[drive]` if none | single canonical source; inherits Phase 9 derived-view discipline |
| direction per (DID, drive) | the `direction` field of that same most recent event; `null` if the fallback baseline path was taken | implicit: baseline is stable, not a transition |
| drive list | frozen `DRIVE_ORDER` constant | closed enum per D-10a-08; ordering never depends on runtime state |
| baseline levels | `DRIVE_BASELINE_LEVEL` compile-time constant | deterministic first-paint; drift-detected against Brain source of truth |

**MUST NOT read from:**
- `fetchNousState()` — that endpoint is Brain-private and must NOT expose
  drive floats per DRIVE-05. The dashboard-side Drives panel never adds a
  `drives` or `ananke` field to the `NousStateResponse` TypeScript shape.
  (Type-level guard: a grep test over `dashboard/src/lib/api/introspect.ts`
  asserts zero occurrences of the `DRIVE_FORBIDDEN_KEYS` set; planner wires
  this grep.)
- Any new HTTP endpoint. Phase 10a ships zero new fetch paths on the
  dashboard side.
- Any new WebSocket subscription. 10a reuses the existing audit stream.
- Brain RPC. The dashboard has no Brain RPC client; all inner-state views
  flow through the Grid's audit stream.

This is the structural privacy guarantee: there is no code path from the
dashboard into drive floats because the only transport carries the bucketed
`level` enum already.

## Privacy Contract (Render-Surface Enforcement)

**The Drives panel MUST NOT render any numeric drive value anywhere in the
DOM.** This is the UI-side declaration of DRIVE-05 and D-10a-07. Enforcement
is delegated to planner via grep CI gates, but the UI-SPEC names the surfaces:

| Surface | Enforcement rule |
|---------|------------------|
| Visible text nodes | Never contain a float that could be a drive value; the only numbers the panel renders are none in 10a — there are no counters, percentages, priorities, or tick numbers in a drive row. Drive level is encoded as the enum text `low` / `med` / `high` |
| `title` attribute | Never set on drive rows or drive glyphs. If a future tooltip ships, its text is also enum-valued. |
| `aria-label` | Encodes the level enum (not a float) + drive name + optional direction. Exact strings: see §Accessibility Matrix below. |
| `data-*` attributes | `data-drive`, `data-level`, `data-direction` each hold enum values (not floats). Never a `data-value`. Never a `data-drive-raw`. |
| Render-surface grep gate | Planner adds `dashboard/test/privacy/drive-forbidden-keys.test.ts` asserting DRIVE_FORBIDDEN_KEYS (`{hunger, curiosity, safety, boredom, loneliness, drive_value}`) absent from any `title=`, `aria-label=`, and literal string in rendered ananke output. (Exact grep pattern is planner's; the UI-SPEC declares the surface set.) |

The dashboard privacy grep tests are the enforcement layer. This UI-SPEC is
the design-side declaration that the executor and checker treat as the
contract. If a future refactor introduces a drive-float render path, the UI
rule here is broken first and the test gate catches it second — the two are
belt and suspenders.

## Accessibility Matrix

**Axis:** 5 drives × 3 levels × 3 directions (rising, falling, stable=null) =
45 distinct semantic states. Each has a deterministic `aria-label` string.

### Row-level aria-label template

```
`{drive} level {level}{, {direction}}`
```

Where:
- `{drive}` is one of `hunger | curiosity | safety | boredom | loneliness`
- `{level}` is one of `low | med | high`
- `{direction}` is `rising` or `falling` — **omitted entirely** (including the comma) when the drive is stable

### 45-case matrix (aria-label on the `<span data-testid="drive-level-{drive}">`)

The label is attached to the level span rather than the `<li>` row so a
screen-reader reading the row in order hears: *drive glyph (hidden) →
drive name (plain) → level label (aria-labelled with full phrase) → direction
glyph (hidden)*.

| Drive | Level | Direction | aria-label |
|-------|-------|-----------|------------|
| hunger | low | stable | `hunger level low` |
| hunger | low | rising | `hunger level low, rising` |
| hunger | low | falling | `hunger level low, falling` |
| hunger | med | stable | `hunger level med` |
| hunger | med | rising | `hunger level med, rising` |
| hunger | med | falling | `hunger level med, falling` |
| hunger | high | stable | `hunger level high` |
| hunger | high | rising | `hunger level high, rising` |
| hunger | high | falling | `hunger level high, falling` |
| curiosity | low..high × stable/rising/falling | (same pattern) | `curiosity level {level}[, {direction}]` |
| safety | low..high × stable/rising/falling | (same pattern) | `safety level {level}[, {direction}]` |
| boredom | low..high × stable/rising/falling | (same pattern) | `boredom level {level}[, {direction}]` |
| loneliness | low..high × stable/rising/falling | (same pattern) | `loneliness level {level}[, {direction}]` |

**Why the pattern is declared once, not enumerated 45 times:** the template is
the contract; the test generates the 45 assertions from `DRIVE_ORDER × LEVELS ×
DIRECTIONS + stable` to avoid drift between spec and test.

### List-level aria

- The `<ul>` carries `role="list"` (Tailwind 4's CSS reset removes list
  semantics from raw `<ul>` without an explicit `role`) and `aria-label="Current
  drive pressure levels"`.
- The `<section>` carries `aria-labelledby="section-ananke-title"` pointing at
  the `<h3 id="section-ananke-title">Drives</h3>`.

### Keyboard

- Rows are **not** focusable in Phase 10a. There are no click handlers, no
  expand/collapse, no elevation triggers. `tabIndex` is unset; default is
  `-1` for `<li>`. Rows receive keyboard focus only if a future phase adds
  affordances (not planned).
- The overall inspector tab-focus flow (Psyche → Thymos → **no-op in Ananke**
  → Telos-refined-badge button → Memory) is unchanged.

### Color-not-sole-channel

Each level is encoded by **both** color (dot + text) and text (the level
label `low`/`med`/`high`). Each drive is encoded by **both** glyph (shape) and
text (drive name). Direction is encoded by **both** arrow glyph and aria-label
clause. A low-vision or monochrome operator can read all 45 semantic states
without color.

### Contrast

All text/background pairs listed in §Level Bucket Color Encoding meet WCAG 2.1
AA for small body text. No additional contrast verification is needed beyond
the Phase 9 human-verify checkpoint already in the record.

| Drive row label — safety | `safety` | same |
| Drive row label — boredom | `boredom` | same |
| Drive row label — loneliness | `loneliness` | same |
| Level label — low | `low` | Lowercase; matches payload enum `level` field. |
| Level label — med | `med` | Lowercase; abbreviated per the DRIVE-03 payload enum (not `medium`). |
| Level label — high | `high` | Lowercase; matches payload enum `level` field. |
| Direction — rising | `↑` visually; `rising` in aria | The aria text is locked at `rising`, matching the payload enum `direction` field. |

| Empty state | **none — panel has no empty state** | By design: the panel always renders five rows at baseline, even for a Nous with no prior crossings. This is the deterministic "current tier read-out" contract (§Baseline). There is no "waiting for event" placeholder, no `<EmptyState>`, no spinner. |
| Loading state | **none — panel has no loading state** | The audit stream is already live when the Inspector mounts; there is no fetch lifecycle specific to Ananke. If the audit stream itself is offline, the Inspector-level error states (Phase 4) handle it; Ananke does not add a second error surface. |
| Error state | **none — panel has no error state** | Same reason. Errors at the audit-stream layer are surfaced by the existing dashboard transport code. |
| Primary CTA | **none — panel has no actions** | DRIVE-04: drives are advisory only. No "elevate pressure", no "override", no "force action". The panel is read-only. |
| Destructive confirmation | **none — no destructive actions in 10a** | Follows DRIVE-04 + PHILOSOPHY §6. |

**Why no empty state / loading / error:** the panel's read model guarantees a
determined first-paint state (baseline rows). Introducing an empty/loading/
error path would add UI states that never fire in normal operation and would
create a surface where the panel could "forget" to render the current drive
state. The deterministic-first-paint discipline is a defensive pattern, not an
ergonomic shortcut.

## Testing Contract (for planner/executor)

The UI-SPEC names the test obligations; the planner writes the specific test
file structure.

1. **45-case aria-label matrix** — one Vitest assertion per (drive × level ×
   direction) combination + 15 stable-direction combinations, generated from
   `DRIVE_ORDER × LEVELS × DIRECTIONS`.
2. **Baseline-first-paint test** — Inspector mounts on a DID with zero
   `ananke.drive_crossed` events; 5 rows render; each row shows `DRIVE_BASELINE_LEVEL[drive]` label; no `direction` span in any row.
3. **Transition test** — simulate an `ananke.drive_crossed` event (`hunger, low→med, rising`) landing in the audit store; assert the hunger row updates to
   `med` label + amber dot + `↑` glyph.
4. **Color-not-sole-channel test** — with color disabled via forced
   `filter: grayscale(1)` (or equivalent vitest-friendly assertion), every
   level label is still distinguishable via its text content.
5. **No-numeric-drive-render test** — the `DRIVE_FORBIDDEN_KEYS` regex asserts
   absence over the full rendered panel DOM (including `title`, `aria-label`,
   `data-*` attributes). Planner wires the exact test shape.
6. **No-wall-clock test** — grep `dashboard/src/app/grid/components/inspector-sections/ananke.tsx` + its hook for `Date.now | performance.now | setInterval | setTimeout` (the direction-glyph window is pure audit-stream-derived, never timer-based).
7. **Drift-detector test** — `dashboard/test/lib/ananke-types.drift.test.ts`
   reads the Brain source of truth for baseline constants and fails on divergence.
8. **Section-ordering test** — within the Overview tabpanel, `section-ananke`
   appears between `section-thymos` and `section-telos` in DOM order.
9. **Inspector integration test** — existing `inspector.test.tsx` assertions
   extend to include the new section's presence when `selectedDid` is non-null
   and `state.status === 'ok'`.
