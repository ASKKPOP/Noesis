Inherits the dashboard neutral-950 dark palette. Phase 8 introduces rose/red as a new semantic destructive accent layer — not used by any prior surface.

| Role | Tailwind Token | Hex Approx | Usage |
|------|---------------|------------|-------|
| Dominant surface (60%) | `bg-neutral-950` | #0A0A0A | Inspector drawer bg; dialog bg; body bg |
| Secondary surface (30%) | `bg-neutral-900` | #171717 | Button secondary fill; ElevationDialog bg (`elevation-dialog.tsx:63`); inner code block bg |
| Tertiary / border | `border-neutral-800` | #262626 | Dividers, drawer border-l, button borders, dialog frame default |
| Text — primary | `text-neutral-100` | #F5F5F5 | Drawer headings, dialog h2, primary body text |
| Text — secondary | `text-neutral-400` | #A3A3A3 | Inspector section labels, dialog body copy, payload preview |
| Text — tertiary / muted | `text-neutral-600` | #525252 | Disabled affordance text, Inspector footer, caption copy |
| **Destructive accent (10%)** | `border-rose-900` / `border-red-600` | #881337 / #DC2626 | **RESERVED FOR:** IrreversibilityDialog outer border only |
| **Destructive text** | `text-rose-400` / `text-red-500` | #FB7185 / #EF4444 | **RESERVED FOR:** first-life promise warning paragraph; Delete button label |
| **Destructive button ring** | `ring-rose-500` | #F43F5E | Focus ring on Delete button only |
| **Destructive chip** | `text-red-400 line-through` | #F87171 | Firehose `operator.nous_deleted` actor name rendering |
| Focus — neutral | `ring-neutral-500` | #737373 | Focus ring on Cancel, Inspector close, all non-destructive controls |
| Code block text | `text-neutral-200` | #E5E5E5 | `<code>` DID element inside dialog |
| Code block bg | `bg-neutral-800` | #262626 | `<code>` element background |

**60/30/10 split:**
- 60% `bg-neutral-950` — all surface areas
- 30% `bg-neutral-900` + `border-neutral-800` — secondary UI chrome, button fills, code bg
- 10% rose/red — exclusively destructive vocabulary: dialog border, warning copy, Delete button, firehose tombstone chip

**Accent reserved for — explicit list:**
1. IrreversibilityDialog outer border: `border-2 border-red-600` (or `border-rose-900` — see Tailwind note below)
2. First-life promise paragraph text: `text-rose-400`
3. Delete Forever button label: `text-red-500` (disabled state: `text-red-500/40`)
4. Delete Forever button focus ring: `ring-2 ring-rose-500`
5. Firehose row actor column when `eventType === 'operator.nous_deleted'`: `text-red-400 line-through`
6. Firehose row event-type badge for `operator.nous_deleted`: treated as `lifecycle` category (`bg-neutral-500/10 text-neutral-300`) — overridden with `bg-rose-900/20 text-rose-300` to mark destruction distinctly

**Tailwind token note:** `border-red-600` (#DC2626) is the Phase 6 H4 tier border color
(`border-2 border-red-400` from chip.tsx `red` variant). To distinguish IrreversibilityDialog
from the H4 chip, use `border-rose-900` (#881337) for the dialog frame — darker, unmistakably
different from the H4 red chip affordance. Delete button label uses `text-red-500` (slightly
brighter than the H4 chip's `text-red-400`) to maintain contrast. These two surfaces (H4 chip
vs H5 dialog) will never appear simultaneously — but the token distinction prevents visual
ambiguity if they do.

**WCAG 2.1 AA compliance:**
- `text-rose-400` (#FB7185) on `bg-neutral-950` (#0A0A0A): contrast ~6.3:1 — passes AA (4.5:1 threshold).
- `text-red-500` (#EF4444) on `bg-neutral-950` (#0A0A0A): contrast ~4.6:1 — passes AA at 12px+ (text-xs threshold satisfied).
- `text-neutral-400` (#A3A3A3) on `bg-neutral-950` (#0A0A0A): contrast ~7.2:1 — passes AA.
- `text-neutral-600` (#525252) on `bg-neutral-950` (#0A0A0A): contrast ~3.8:1 — FAILS AA for small text; acceptable for `text-[10px]` meta captions only (decorative / supplemental, not instructional).
- All instructional copy (warning, label, DID display) must be `text-neutral-400` or higher.

## Component Inventory

### Surface 1: Inspector H5 Delete Button — Active State

**File:** `dashboard/src/app/grid/components/inspector.tsx` (edit lines 219–234)

**Current state (Phase 6 D-20):**
```tsx
<button
  disabled
  aria-disabled="true"
  title="Requires Phase 8"
  className="w-full cursor-not-allowed rounded border border-neutral-800 bg-neutral-900 px-3 py-2 text-xs text-neutral-600 line-through"
>
  Delete Nous
</button>
```

**Phase 8 target — three rendering states:**

#### State A: Active (operator tier H1–H4, Nous is alive)
```tsx
<button
  type="button"
  data-testid="inspector-h5-delete"
  onClick={handleDeleteClick}
  className="w-full rounded border border-neutral-800 bg-neutral-900 px-3 py-2 text-xs text-neutral-400 hover:border-rose-900 hover:text-rose-400 focus:outline-none focus:ring-2 focus:ring-rose-500 transition-colors"
  aria-label="Delete this Nous permanently (H5 Sovereign required)"
>
  Delete Nous
</button>
```

**Interaction contract:**
- `cursor-default` (implicit; no `cursor-not-allowed`)
- Hover: `border-neutral-800` → `border-rose-900`, `text-neutral-400` → `text-rose-400`
- Focus ring: `ring-2 ring-rose-500 ring-offset-1 ring-offset-neutral-950` (2px ring, 1px offset on dark bg)
- Transition: `transition-colors duration-150`
- Click: opens `ElevationDialog` with `targetTier="H5"` — see Surface 2 / two-stage flow
- No `disabled` attribute; no `aria-disabled`; tabIndex default (0)
- Caption below button: `<p className="mt-1 text-[10px] text-neutral-500">H5 — irreversible. Audit entries remain.</p>` (replaces Phase 6 caption)

#### State B: Already tombstoned (`state.data.status === 'deleted'`)
```tsx
<button
  type="button"
  data-testid="inspector-h5-delete"
  disabled
  aria-disabled="true"
  className="w-full cursor-not-allowed rounded border border-neutral-800 bg-neutral-900 px-3 py-2 text-xs text-neutral-600"
>
  Nous deleted at tick {state.data.deletedAtTick ?? '—'}
</button>
```

**Interaction contract:**
- Non-interactive; `cursor-not-allowed`
- No hover effect
- Label reads "Nous deleted at tick N" (no line-through; the button is now informational, not suppressed)
- Caption below: `<p className="mt-1 text-[10px] text-neutral-600">Audit history available in the firehose.</p>`

#### State C: Loading / error / idle (Nous state unknown)
- Render state A (active) with onClick guarded: if `state.status !== 'ok'`, clicking does nothing (guard inside `handleDeleteClick` before opening elevation dialog). This prevents attempting H5 on a DID we cannot confirm is alive.
- Button appearance: same as State A but clicking shows no elevation dialog; a no-op until introspect resolves.
- Alternative: hide button entirely during loading — **do not hide**. PHILOSOPHY §7 mandates the tier always be visible. Render active appearance, guard the click handler.

**Rendering rule:** The `status` field is only available when `state.status === 'ok'` (the introspect returned successfully). When `state.status === 'ok' && state.data.status === 'deleted'`, render State B. Otherwise render State A.

### Surface 2: Two-Stage Elevation Flow

**Trigger:** click of inspector-h5-delete in State A.

#### Stage 1 — ElevationDialog (existing Phase 6 primitive)

**File:** `dashboard/src/components/agency/elevation-dialog.tsx` (no changes)

The existing primitive is invoked with `targetTier="H5"`. Since `ElevationDialog`'s `ElevatedTier` type excludes H5 (`Exclude<HumanAgencyTier, 'H1' | 'H5'>`), Phase 8 must either:
- Extend `ElevatedTier` to include `'H5'` and add `CONFIRM_FILL['H5']` = `'bg-rose-700 text-neutral-100'`
- OR use a custom H5 elevation inline before the IrreversibilityDialog

**Decision (conservative default per --auto):** Extend `ElevatedTier` to include `'H5'` and add the H5 confirm fill. H5 confirm button fill: `bg-rose-700 text-neutral-100`. This keeps the two-stage flow consistent (ElevationDialog always gates entry to any elevated tier) and avoids an ad-hoc non-primitive elevation path.

ElevationDialog copy for H5: `"Entering H5 — Sovereign. This will be logged."` (matches existing format `Entering ${targetTier} — ${tierName}. This will be logged.` where `TIER_NAME['H5'] = 'Sovereign'`).

**On confirm:** agencyStore sets tier to H5, ElevationDialog closes, IrreversibilityDialog opens.
**On cancel:** flow aborts, auto-downgrade to H1 (Phase 6 D-08 pattern).

**Confirmation input:**
```tsx
<input
  id="irrev-did-input"
  type="text"
  autoComplete="off"
  autoCorrect="off"
  autoCapitalize="off"
  spellCheck={false}
  data-testid="irreversibility-did-input"
  value={typed}
  onChange={(e) => setTyped(e.target.value)}
  onPaste={(e) => e.preventDefault()}
  aria-label="Type the full DID to confirm deletion"
  aria-invalid={typed.length > 0 && typed !== targetDid ? 'true' : undefined}
  aria-describedby="irrev-input-hint"
  className={[
    'w-full rounded border bg-neutral-900 px-3 py-2 font-mono text-xs text-neutral-100',
    'placeholder:text-neutral-600',
    'focus:outline-none focus:ring-2 focus:ring-offset-1 focus:ring-offset-neutral-950',
    typed.length === 0
      ? 'border-neutral-700 focus:ring-neutral-500'
      : typed === targetDid
        ? 'border-neutral-700 focus:ring-neutral-500'
        : 'border-rose-900 focus:ring-rose-500',
  ].join(' ')}
/>
```
- `onPaste={(e) => e.preventDefault()}` — D-01 paste suppression; forces character-by-character transcription
- `autoComplete="off" autoCorrect="off" autoCapitalize="off" spellCheck={false}` — prevents browser assistance that could bypass transcription intent
- `aria-invalid` fires only when user has started typing AND it doesn't match — avoids announcing invalid on an untouched field
- Border color: neutral-700 (idle + correct), rose-900 (typing mismatch) — red border as inline validation signal
- Input font: `font-mono text-xs` — matches DID display font so operator can visually align each character

**Validation hint (conditional):**
```tsx

  className="mt-1 min-h-[16px] text-[11px]"

```
- `min-h-[16px]` — reserves space so the button row doesn't jump when hint appears/disappears
- `role="status"` — polite live region; SR announces the hint on change without interrupting

**Button row:**
```tsx
<div className="mt-5 flex justify-end gap-2">

    Cancel

    type="button"

    onClick={handleConfirm}

      'focus:outline-none focus:ring-2 focus:ring-rose-500',

    aria-disabled={!exact ? 'true' : undefined}

    Delete forever
  </button>
</div>
```

**Button interaction contract:**
- Cancel: `autoFocus` — D-04/D-05; safer default so Enter-on-dialog-open cannot trigger delete
- Cancel hover: `text-neutral-200` → `text-neutral-100`; no background change
- Delete forever: only enabled when `typed === targetDid` (exact case-sensitive match)
- Delete forever enabled: `text-red-500`, hover `text-red-400`; focus ring `ring-rose-500`
- Delete forever disabled: `text-red-500/40`, `cursor-not-allowed`; `aria-disabled="true"`
- **No `<form>` submit path** — D-03: `onSubmit={(e) => e.preventDefault()}` on any wrapping form element; Delete is `type="button"` with explicit onClick only

**Focus trap:** Native `<dialog>` + `showModal()` provides browser-native focus trap (D-08 Phase 6 pattern). No hand-rolled trap needed. ESC fires native `close` event which routes to `onCancel`.

**ESC handling:**
```tsx
<dialog onClose={handleCancel} ...>
```
The native `close` event fires on both ESC and `dialogRef.current.close()`. `handleCancel` auto-downgrades tier to H1 (D-05).

**Focus restoration:** On dialog close (any path), restore focus to the Inspector close button (`firstFocusable.current` in inspector.tsx:75 — `data-testid="inspector-close"`). This is the openerRef pattern from Phase 6 D-07. Capture the opener ref at IrreversibilityDialog open time:
```tsx
// Inside Inspector, before opening IrreversibilityDialog:
const irrevOpenerRef = useRef<HTMLElement | null>(null);
// At open time:
irrevOpenerRef.current = document.activeElement as HTMLElement | null;
// At close time (handleCancel / handleConfirm):
irrevOpenerRef.current?.focus();
```

**Closure-capture race safety (D-22):** `targetDid` is captured at dialog-open time from the Inspector's `selectedDid`. If `selectedDid` changes while the dialog is open (edge case), the dialog still commits against the DID the operator saw. Achieved by passing `targetDid` as a prop at mount time — not reading from store/hook at confirm time.

```tsx
  'hover:bg-neutral-900/50',

    ? 'border-l-2 border-l-rose-900'
    : 'hover:border-l hover:border-l-sky-300',
  dimClass,
].join(' ');
```
- Tombstone rows get a permanent `border-l-2 border-l-rose-900` left accent (no hover variant needed — always visible)
- Other rows retain the existing `hover:border-l hover:border-l-sky-300` behavior

### Surface 6: SR / Live Region Announcements

**File:** Root layout or a new `aria-announcer.tsx` primitive under `dashboard/src/components/accessibility/`

Phase 8 requires three announcement categories:

| Moment | Copy | Region Role | Politeness |
|--------|------|-------------|------------|
| Dialog opens, state-hash computation begins | "Computing pre-deletion state hash…" | `aria-live="polite"` | polite |
| Delete route returns 200 | "Nous {name} deleted successfully." | `role="status"` | polite |
| Delete route returns 503 (brain unavailable) | "Deletion failed: the Brain is unreachable. Try again." | `role="alert"` | assertive |
| Delete route returns 410 (already deleted) | "This Nous was already deleted." | `role="status"` | polite |
| Delete route returns 400 (bad DID) | "Deletion failed: invalid Nous ID." | `role="alert"` | assertive |

**Implementation pattern:** A single `<div aria-live="polite" aria-atomic="true" className="sr-only" data-testid="aria-announcer" />` in the root layout, updated imperatively from an `announcer` module. This follows the existing Phase 6 pattern where the AgencyIndicator is `fixed` in the layout — the announcer is its SR complement.

For assertive (error) announcements, the same element temporarily switches to `aria-live="assertive"` via a React key flush trick, or a second assertive region is maintained. Conservative default: two regions (`polite-announcer` + `assertive-announcer`), both `sr-only`.

- Close IrreversibilityDialog.
- Show toast with error copy.
- Agency auto-downgrades to H1.

### Keyboard Navigation Contract

| Key | Context | Action |
|-----|---------|--------|
| Tab | Inspector open | Cycle through focusable elements; Delete button included when enabled |
| Shift+Tab | Inspector open | Reverse cycle |
| Escape | Inspector open (no dialog) | Close Inspector (`selection.clear()`), restore focus to opener |
| Escape | ElevationDialog open | Close dialog, auto-downgrade H1 (native `<dialog>` `close` event) |
| Escape | IrreversibilityDialog open | Close dialog, auto-downgrade H1 (native `<dialog>` `close` event) |
| Enter | IrreversibilityDialog open | No submit (D-03: `onSubmit` prevented; Delete is `type="button"`) — cursor stays in input |
| Tab | IrreversibilityDialog open | Browser-native focus trap cycles: Cancel → Delete → input → Cancel |
| Space | Cancel button focused | Activate Cancel |
| Space | Delete button focused (enabled) | Activate Delete |

**Focus order inside IrreversibilityDialog:**
1. Cancel button (`autoFocus`) — always first, per D-04 safer default
2. Delete button (when enabled; when disabled, `disabled` attribute removes from natural tab order)
3. Input (`id="irrev-did-input"`)

Wait — an input should be focusable before buttons for form usability. However, D-04 specifies Cancel gets `autoFocus` (same pattern as ElevationDialog Cancel). The operator must first read the dialog before typing. After autoFocus on Cancel, operator presses Tab to reach the input, then types, then Tab to Delete button.

**Revised focus order:**
1. Cancel button (autoFocus — initially focused to prevent accidental Enter-to-delete)
2. Input field
3. Delete button (enabled only after exact match)

The native focus trap keeps this cycle within the dialog boundaries.
