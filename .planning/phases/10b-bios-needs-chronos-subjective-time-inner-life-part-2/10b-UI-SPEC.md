## Scope Boundary

Phase 10b ships **exactly one UI surface**: a new section component
`<BiosSection did={did} />` mounted inside the Inspector Overview tab as the
6th sibling in render order — immediately after `<AnankeSection>` (Phase 10a)
and before `<TelosSection>`. It is not a new page, route, modal, or
design-system shift. No new global tokens, no new primitives, no new
dependencies.

**In scope:**
- Panel layout within `inspector.tsx` Overview tab — one new section between AnankeSection and TelosSection
- 2 need rows × 3 level buckets × 2 direction indicators (rising/falling) rendered as Unicode + color, cloning AnankeSection row vocabulary exactly
- Accessibility contract: per-need aria-label matrix; 9 distinct semantic states (3 levels × energy + 3 levels × sustenance + 3 rest/stable variants per need); color-not-sole-channel compliance
- Baseline-level initial state when no `ananke.drive_crossed` (hunger / safety) has landed for the DID — energy defaults low, sustenance defaults low
- Zero-numerics guarantee (D-10b-10 BIOS_FORBIDDEN_KEYS enforcement at the render surface)
- Source-of-truth contract: panel reads from the existing `ananke.drive_crossed` broadcast stream filtered to `hunger` (energy→hunger) and `safety` (sustenance→safety); no new fetch, no new endpoint
- Two new glyph constants: ⚡ for energy (U+26A1 LIGHTNING SIGN) and 🜃 fallback ∿ for sustenance (U+223F SINE WAVE used as food-neutral symbol; see §Need Glyph Matrix)
- SYNC mirror for Bios baseline constants in `dashboard/src/lib/protocol/bios-types.ts` (drift-detected against Brain `bios/config.py`)

**Out of scope — Chronos (deferred, zero UI in 10b):**
- Subjective-time multiplier surface (D-10b-05: Brain-local read-side state only)
- Chronos panel, tooltip, or badge of any kind
- Any UI that would surface `CHRONOS_FORBIDDEN_KEYS` (`subjective_multiplier`, `chronos_multiplier`, `subjective_tick`)
- Rationale from CONTEXT.md §specifics: "Chronos is invisible on the Dashboard in 10b. Subjective time is an internal cognitive mechanism; operator surfacing is deferred."

**Out of scope — other:**
- Numeric need-float rendering anywhere in DOM, tooltip, title, or aria text (anti-feature per D-10b-10)
- Need time-series charts / sparklines (roadmap defers quantitative views)
- Bios → Ananke elevation coupling UI (elevation is Brain-internal per D-10b-02; no UI surface implies a need "caused" a drive change)
- Mobile / responsive adaptations (dashboard is desktop-only per existing convention)
- New route, new entry in `DASHBOARD_ROUTES` E2E SC#1 coverage list
- New icon library, new SVG assets, or new third-party components

## Panel Layout

### Position in the Inspector drawer

```
Inspector drawer (fixed right, 384px wide, dark neutral-950)
├── header: "Nous Inspector" + close
├── tab strip: [ Overview | Relationships ]
│
├── Overview tabpanel (hidden when tab=relationships)
│     ├── <PsycheSection>         ── Phase 4 / v2.0
│     ├── <ThymosSection>         ── Phase 4 / v2.0
│     ├── <AnankeSection>         ── Phase 10a (Drives)
│     ├── <BiosSection>    NEW    ── Phase 10b ◄◄◄ THIS PHASE
│     ├── <TelosSection>          ── Phase 7 refinement badge
│     └── <MemorySection>         ── Phase 4 / v2.0
│
└── Relationships tabpanel (Phase 9; unchanged by 10b)
      └── <RelationshipsSection>
```

**Mount position rationale:**
- Drives (Ananke) → **Needs (Bios)** → Goals (Telos) forms the inner-life
  causal chain: drive pressure rises → bodily need escalates the matching drive
  → goal orientation adapts. Bios immediately follows Ananke because Bios
  elevation feeds back into the Ananke drive row above it — visual adjacency
  makes the energy→hunger and sustenance→safety couplings legible without
  implying that the panel itself shows causation.
- Placing Bios before Telos maintains the Psyche → Thymos → Ananke → Bios →
  Telos → Memory personality-to-history ordering established by 10a-UI-SPEC.

**Need ordering (locked):** `NEED_ORDER = ['energy', 'sustenance']`. Matches
BIOS-01 enumeration order. Order is stable across ticks.

## Need Glyph Matrix

Unicode glyphs only. Exactly 2 new glyph constants introduced in this phase;
all other glyph/color constants are direct clones or imports from 10a.

Each glyph is:
- a single code point distinguishable from the 10a drive glyphs at 14px/`text-sm` with `leading-none`
- legible on `neutral-900` background
- semantically suggestive of the need without depending on color
- present in the system default emoji/symbol fonts across macOS / Linux / Windows

| Need | Glyph | Code point | Semantic cue | Fallback text (if glyph fails to render) |
|------|-------|-----------|--------------|------------------------------------------|
| energy | ⚡ | U+26A1 (HIGH VOLTAGE SIGN) | electrical charge / depletion | `energy` |
| sustenance | ⬡ | U+2B21 (WHITE HEXAGON) | container / vessel awaiting fill | `sustenance` |

**Fallback rationale for sustenance:** U+2B21 WHITE HEXAGON is chosen as a
food-neutral abstract vessel — a hexagonal container shape — avoiding
culturally-specific food emoji (🍎, 🥣) that carry connotation inappropriate
for an abstract Nous. If U+2B21 renders as a tofu box on any test environment,
the fallback is the literal text `sustenance` (following the same fallback
discipline as 10a-UI-SPEC §Drive Glyph Matrix). Planner executes a visual-smoke
screenshot on macOS + Linux + Windows during phase verification.

**No color channel on the need glyph.** The glyph sits in `text-neutral-200` at
all levels. Level encoding is handled by the dot + text-label color pair.
Same double-encoding discipline as 10a (shape=need, color=level, orthogonal axes).

## Level Bucket Color Encoding

**Identical to 10a-UI-SPEC §Level Bucket Color Encoding.** No new tokens.
Reuses the neutral-amber-rose ramp Phase 9 established and Phase 10a adopted.

| Level | Semantic | Dot class | Text class | Rationale |
|-------|----------|-----------|-----------|-----------|
| `low` | baseline / quiescent | `bg-neutral-400` (#9CA3AF) | `text-neutral-400` | Neutral conveys "nothing to act on" |
| `med` | elevated / active | `bg-amber-400` (#FBBF24) | `text-amber-400` | Amber conveys "pay attention but not alarm" |
| `high` | pressing / salient | `bg-rose-400` (#FB7185) | `text-rose-400` | Rose conveys "this need is loud right now" |

Contrast ratios (inherited from 10a-UI-SPEC, human-verified Phase 9):
- `text-amber-400` on `bg-neutral-900` → 9.8:1 (AAA large, AA small)
- `text-rose-400` on `bg-neutral-900` → 6.9:1 (AA large, AA small)
- `text-neutral-400` on `bg-neutral-900` → 5.7:1 (AA large, AA small)

All three meet WCAG 2.1 AA for small text.

## Source-of-Truth Contract

The Bios panel reads **indirectly from the existing `ananke.drive_crossed` audit
stream**, not from a new Bios-specific event. This is the disciplined mapping
from BIOS-01: energy → hunger, sustenance → safety. No new fetch, no new
endpoint.

| Read | Where from | Why |
|------|-----------|-----|
| current level for energy | most recent `ananke.drive_crossed` where `actorDid === did && payload.drive === 'hunger'`; fallback to `NEED_BASELINE_LEVEL['energy']` | D-10b-02: Bios elevation surfaces via existing `ananke.drive_crossed`; energy → hunger is the canonical mapping |
| current level for sustenance | most recent `ananke.drive_crossed` where `actorDid === did && payload.drive === 'safety'`; fallback to `NEED_BASELINE_LEVEL['sustenance']` | D-10b-02: sustenance → safety mapping |
| direction per (DID, need) | the `direction` field of that same most recent event; `null` if the fallback baseline path was taken | same as 10a |
| need list | frozen `NEED_ORDER` constant `['energy', 'sustenance']` | closed enum per BIOS-01 |
| baseline levels | `NEED_BASELINE_LEVEL` compile-time constant in `bios-types.ts` | deterministic first-paint; drift-detected against Brain `bios/config.py` |

**MUST NOT read from:**
- `fetchNousState()` — the endpoint MUST NOT expose need floats per D-10b-10.
  The `NousStateResponse` TypeScript shape must remain free of any Bios field
  key. Grep test asserts `BIOS_FORBIDDEN_KEYS` absent from `dashboard/src/lib/api/introspect.ts`.
- Any new HTTP endpoint. Phase 10b ships zero new dashboard fetch paths.
- Any new WebSocket subscription. Reuses existing audit stream.
- Brain RPC. No Brain RPC client in dashboard.

**Why this approach is correct:** Bios elevates Ananke drives (not vice versa),
and the elevation surfaces via `ananke.drive_crossed`. Filtering that stream by
the two mapped drives (hunger / safety) gives the operator a real-time read of
Bios-elevated pressure without adding a new event type, without breaking the
19-event (growing to 21 via `bios.birth`/`bios.death`) allowlist freeze, and
without needing a new consumer API. The Bios panel is a windowed view over
existing data.

## Baseline / Initial State (First-paint contract)

When the Inspector opens on a DID for which no `ananke.drive_crossed` event for
`hunger` or `safety` has yet landed (fresh Nous just past `bios.birth`, or
a replay before any crossing), the panel renders two rows at **per-need
baseline level**.

**Baseline mirror format (locked by planner):**

```ts
// dashboard/src/lib/protocol/bios-types.ts
// SYNC: mirrors brain/src/noesis_brain/bios/config.py NEED_BASELINES (bucketed)
// Drift is detected by dashboard/test/lib/bios-types.drift.test.ts
// which reads the Python source and fails if values diverge.
export const NEED_BASELINE_LEVEL: Record<NeedName, DriveLevel> = {
    energy: 'low',      // brain baseline 0.3 → bucket low (threshold 0.33)
    sustenance: 'low',  // brain baseline 0.3 → bucket low (threshold 0.33)
};

export const NEED_ORDER: NeedName[] = ['energy', 'sustenance'];

export const NEED_GLYPH: Record<NeedName, string> = {
    energy:     '\u26A1',  // ⚡ U+26A1 HIGH VOLTAGE SIGN
    sustenance: '\u2B21',  // ⬡ U+2B21 WHITE HEXAGON
};

// Drive mapping for useBiosLevels hook: need → which drive to watch
export const NEED_TO_DRIVE: Record<NeedName, DriveName> = {
    energy:     'hunger',
    sustenance: 'safety',
};
```

**The dashboard never stores or renders raw need floats.** Baseline floats are
used only at module-load time to compute the bucketed `DriveLevel` constant.
The floats themselves never enter rendered DOM. Grep test asserts no literal
`0.3` (or any `0\.[0-9]+`) in rendered Bios panel output — same discipline as
10a's drive-baseline grep gate (D-10a-07 extended to BIOS_FORBIDDEN_KEYS).

**Baseline rows render without a direction glyph.** Baseline is a state, not
a transition. The row reads `"energy  low"` with no arrow.

## Accessibility Matrix

**Axis:** 2 needs × 3 levels × 3 directions (rising, falling, stable=null) =
**9 distinct semantic aria-label states per need = 18 total states** in the
panel. Each has a deterministic `aria-label` string.

### Row-level aria-label template

```
`{need} level {level}{, {direction}}`
```

Where:
- `{need}` is one of `energy | sustenance`
- `{level}` is one of `low | med | high`
- `{direction}` is `rising` or `falling` — **omitted entirely** (including the comma) when stable

### 18-case matrix (aria-label on `<span data-testid="need-level-{need}">`)

| Need | Level | Direction | aria-label |
|------|-------|-----------|------------|
| energy | low | stable | `energy level low` |
| energy | low | rising | `energy level low, rising` |
| energy | low | falling | `energy level low, falling` |
| energy | med | stable | `energy level med` |
| energy | med | rising | `energy level med, rising` |
| energy | med | falling | `energy level med, falling` |
| energy | high | stable | `energy level high` |
| energy | high | rising | `energy level high, rising` |
| energy | high | falling | `energy level high, falling` |
| sustenance | low | stable | `sustenance level low` |
| sustenance | low | rising | `sustenance level low, rising` |
| sustenance | low | falling | `sustenance level low, falling` |
| sustenance | med | stable | `sustenance level med` |
| sustenance | med | rising | `sustenance level med, rising` |
| sustenance | med | falling | `sustenance level med, falling` |
| sustenance | high | stable | `sustenance level high` |
| sustenance | high | rising | `sustenance level high, rising` |
| sustenance | high | falling | `sustenance level high, falling` |

**The template is the contract; the test generates all 18 assertions** from
`NEED_ORDER × LEVELS × DIRECTIONS + stable`, mirroring the 10a 45-case pattern
at smaller scale.

### List-level aria

- The `<ul>` carries `role="list"` and `aria-label="Current bodily need levels"`.
- The `<section>` carries `aria-labelledby="section-bios-title"` pointing at
  the `<h3 id="section-bios-title">Needs</h3>`.

### Keyboard

- Rows are **not** focusable in Phase 10b. No click handlers, no expand/collapse,
  no elevation triggers. `tabIndex` unset. Same contract as AnankeSection.
- The overall inspector tab-focus flow is unchanged. Bios panel is read-only.

### Color-not-sole-channel

Each level is encoded by **both** color (dot + text) and text (the level label
`low`/`med`/`high`). Each need is encoded by **both** glyph (shape) and text
(need name). Direction is encoded by **both** arrow glyph and aria-label clause.
Identical to 10a contract — no new WCAG obligations introduced.
