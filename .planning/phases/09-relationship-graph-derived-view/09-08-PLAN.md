<objective>
Close Gap 2 from 09-VERIFICATION.md (finding ME-02 in 09-REVIEW.md): the H5 operator
inspection route currently accepts shortened edge_key prefixes and resolves them via
`startsWith`, which on prefix collision silently matches the first-inserted edge
and emits `operator.inspected` with a wrong `target_did`. The `|| edgeHash(e) === edgeKey`
branch is dead code (startsWith already covers equality). No test exercises a shortened
edge_key, so the bug is invisible to the suite.

Purpose: Eliminate the silent wrong-edge failure mode by requiring full 64-character
hashes (the edge_hash contract — H1/H2 responses already return full 64-char hashes,
so every legitimate client has what it needs). Remove the dead `||` clause. Add regression
tests that would catch any future regression to prefix matching.

Output:
- Tightened regex `/^[a-f0-9]{64}$/i` at the validation gate
- Strict equality `edgeHash(e) === edgeKey` at the resolver
- Three new test cases in relationships-privacy.test.ts: (a) 16-char edge_key rejected 400,
  (b) 63-char edge_key rejected 400, (c) 64-char full hash succeeds and emits correct
  operator.inspected target_did
</objective>
