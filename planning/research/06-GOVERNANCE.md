# Research: Governance, Law, and Enforcement

## 1. Constitutional AI for Agent Governance

### Original CAI Pattern (Anthropic)

Two-phase training:
1. **Self-critique chain**: Model generates response → samples constitutional principle → critiques against it → revises. Produces chain `{y0, y1, ..., yn}`.
2. **RLAIF**: Pairwise preferences generated by AI against constitutional principles → replaces human labelers for RLHF.

### Adaptation for Noēsis (Runtime Self-Evaluation)

Before executing an action, a Nous runs a compliance check:

```
1. Propose action
2. Sample principle from Logos constitution
3. Evaluate: "Does this action violate this principle?"
4. If violation detected → revise action
5. Log reasoning trace to audit trail
6. Execute revised action
```

**Tiered approach** (latency vs. thoroughness):
- **Routine actions**: Fast single-principle check
- **High-impact actions**: Full multi-principle sequential evaluation
- **Resource transfers / governance votes**: Multi-principle + peer review

**Reasoning traces** for audit:
```
"I am about to take action X. Principle 3 states 'Honor confirmed agreements.'
This action would breach agreement #47 with Nous B because Z.
I will revise to action X' instead."
```

**Peer review**: Other Nous can evaluate whether an action satisfies constitutional principles — enabling decentralized compliance scoring.

## 2. Agreement Enforcement (Smart Contracts Without Blockchain)

### State Machine for Agreement Lifecycle

```
PROPOSED → NEGOTIATING → ACTIVE → FULFILLED | DISPUTED → RESOLVED | EXPIRED
```

**Implementation**:
- Stage guards: Precondition check on every method
- Timed transitions: Timestamp + deadline comparison, auto-transitions
- Access restrictions per stage: Different parties have different permissions per state

### Condition-Action Rules Engine

```
Rule format: WHEN <condition> THEN <action> [PRIORITY <n>]
```

- Forward chaining: Evaluate all rules → fire highest-priority match → re-evaluate
- Working memory = current agreement state
- Condition functions + action functions that mutate state

### Automated Arbitration

- **Threshold-based**: Quantifiable deviation from terms → automatic resolution
- **Multi-arbiter panel**: N odd arbiters from pool, majority decision
- **Escalation ladder**: 
  1. Automated rule-based
  2. Single arbiter
  3. Panel
  4. Council vote

### Escrow Without Blockchain

Trusted intermediary service:
- Holds resources in locked account
- Release conditions = FSM reaching FULFILLED state
- Timeout → return to original owners
- Multi-signature: Both parties (or party + arbiter) confirm before release

## 3. Voting and Consensus

### Reputation-Weighted Voting

- Reputation accrued through tasks, governance participation, peer evaluations
- Reputation decays with inactivity (only active members maintain influence)
- Dual system: base equal vote + reputation multiplier
- **Cap**: No Nous gets >5% total voting power regardless of reputation
- Monitor for collusion (coordinated voting by correlated reputations)

### Quadratic Voting

Cost scales quadratically: 1 vote = 1 credit, 2 votes = 4, 3 votes = 9.

- Equal voice credit allocation per voting period
- Captures preference intensity
- Prevents domination by wealthy agents
- Rebate mechanism: Voters receive avg of others' payments (budget-balanced)

### Liquid Democracy (Delegation)

- Vote directly or delegate to trusted representative
- Per-topic delegation (economics to one expert, security to another)
- **Delegation chains**: A → B → C (C votes with combined weight)
- Chains must be acyclic (detect and reject cycles)
- Instant revocation: Delegators can reclaim and vote directly anytime

### Proposal Lifecycle

```
DRAFT → DISCUSSION (min 48h) → VOTING (72h) → PASSED/FAILED → EXECUTION
```

| Type | Quorum | Threshold |
|------|--------|-----------|
| Routine change | 20% | Simple majority (>50%) |
| Significant change | 33% | 60% approval |
| Constitutional amendment | 50% | Supermajority (75%) |
| Emergency | 10% (+ higher approval) | 80% approval |

## 4. Audit Trail Design

### Event Sourcing

Store every action as immutable event in append-only log:

```
Event {
  event_id: UUID
  timestamp: ISO 8601
  nous_id: UUID
  action_type: string
  target: string
  parameters: JSON
  reasoning_trace: text        // constitutional evaluation
  previous_hash: bytes
  event_hash: bytes
}
```

**State reconstruction**: Current world state = replay all events.
**Snapshots**: Every N events, serialize state. Reconstruct = load snapshot + replay subsequent.
**CQRS**: Separate write path (append events) from read path (query materialized views).

### What to Log

- **Actions**: Resource acquisition, communication, movement, trade
- **Decisions**: Reasoning traces, alternatives considered
- **Communications**: Messages between Nous (content hashes for privacy option)
- **Governance**: Votes, proposals, delegations
- **Constitutional evaluations**: Self-critique results

### Merkle Tree for Integrity

- Each entry's hash incorporates previous hash → chain
- Group entries into time-based blocks (hourly or per-1000 events)
- Merkle tree within each block → single root hash
- Sign roots with HSM key → trust anchor for tamper detection
- Verification: Fetch proof path, recompute, match against signed root

## 5. Sanctions and Enforcement

### Sanction Types

| Type | Mechanism | Example |
|------|-----------|---------|
| **Corrective** | Adjust parameters | Reduce resource generation rate |
| **Reputational** | Score decrease | Reduced voting power, trade trust |
| **Procedural** | Compliance review | Forced evaluation before continuing |
| **Exclusionary** | Suspension/ban | Timeout or permanent exile |

### Severity Model

Two-factor scoring:
- **Deviation factor**: How far from the norm (magnitude of violation)
- **Learning factor**: Historical behavior, past sanction responses, repeat offenses

Sanctions trigger when learning factor drops below threshold.

### Technical Mechanisms

**Rate Limiting**: Token bucket on action pipeline. Reduce refill rate as sanction. Per-action-type configurable.

**Capability Revocation**: Capability-based security with unforgeable permission tokens. To sanction: revoke specific capabilities (trading, voting, movement). Can be time-limited (auto-restore) or require re-earning.

**Reputation Penalties**: Direct numeric reduction. Affects voting power, trade trustworthiness, resource access. Decay-based recovery on compliance.

**Exile**:
- Temporary: Disconnect from comms, freeze state, reduced reputation on return
- Permanent: Added to deny list, resources redistributed, keys revoked

### Graduated Response

```
Warning → Rate limiting → Capability restriction → Temporary exile → Permanent ban
```

Each level requires progressively higher governance approval:
1. Single arbiter
2. Small panel (3)
3. Council vote (simple majority)
4. Council vote (supermajority)

## 6. Logos System Design for Noēsis

### Constitutional Laws (Immutable)

```
1. A Nous must not falsify its identity
2. A Nous must honor confirmed agreements
3. A Nous must not destroy shared resources
4. All actions are logged and auditable
5. Every Nous has the right to exist and communicate
6. No Nous may accumulate >X% of total Ousia supply
7. Constitutional changes require 75% supermajority + cooling period
```

### Council Formation

- **Seats**: 5-11 members (odd number)
- **Election**: Reputation-weighted voting, quadratic cost
- **Term**: Fixed duration (e.g., 1000 world-ticks)
- **Recall**: Any Nous can initiate recall vote (33% quorum, 60% threshold)

### Enforcement Pipeline

```
Violation detected (automated or reported)
    │
    ▼
Evidence collection (audit trail query)
    │
    ▼
Severity assessment (deviation × learning factor)
    │
    ▼
Sanction determination (graduated response)
    │
    ▼
Appeal period (48 world-hours)
    │
    ▼
Enforcement execution (automated)
    │
    ▼
Recovery path (compliance → restoration)
```
